## 1. Vision

**SubKit is the subscription and bill tracker for people who don't want their bank connected.** It's the only one in the category that does cloud sync, real reminders, multi-currency, and bill tracking — without requiring a bank OAuth or a subscription to use it. Free tier holds 5 items forever; Pro ($4.99/mo or $39/yr) unlocks unlimited items, full analytics, and cloud sync across devices.

**The 2-sentence pitch:**
> "SubKit tracks your subscriptions manually — no bank linking — and reminds you before they renew. One number, your monthly total, normalized across every billing cycle. Free for up to 5 subs, Pro for unlimited + multi-device sync."

**The one-line pitch (for marketing):**
> "Your subscriptions, one number you can trust."

## 3. Target user

**Primary persona: "The privacy-conscious multitasker"**
- Age 25-45
- Tech-savvy (comfortable with PWA install, Telegram bots, OAuth flows)
- Has 5-30 active subscriptions (SaaS tools, streaming, AI services, etc.)
- Doesn't trust bank-linked tools (privacy OR security OR both)
- Uses multiple devices (laptop + phone)
- Already pays for a few productivity apps ($5-20/mo range)
- Lives in Telegram / Discord / Slack

**Secondary persona: "The freelancer / small biz owner"**
- 30-50
- Tracks subscriptions as a business expense (separate from personal)
- Wants reminders to keep cash flow predictable
- Multi-currency matters (gets paid in USD, lives in THB or another currency)
- Values invoice attachments for tax season

**Anti-persona: "The Mint user"**
- Wants automatic bank integration
- Wants to see all bank transactions
- Doesn't manually track anything

We don't compete with this anti-persona. If they want SubKit, they don't really want a sub tracker; they want a bank aggregator.

## 6. Technical architecture

### 6.1 Stack (unchanged + additions)

| Layer | Today | v1.0 |
|---|---|---|
| Frontend | Single `index.html` with inline CSS+JS | Same (refactor to ESM modules is a v2.0 task) |
| Hosting | Vercel static | Same |
| Auth | Supabase email + GitHub + Google OAuth | Same + Stripe customer creation on signup |
| Database | Supabase Postgres + RLS | Same + new `users` + payment tables |
| Storage | Supabase Storage (`invoices` bucket) | Same + 50 MB limit for Pro |
| Edge functions | `send-reminders` | + `stripe-webhook` + `create-checkout` |
| Reminders | EmailJS (browser) + Telegram (browser) + Edge function | Same |
| Payments | None | Stripe Checkout (hosted) + Webhooks |
| Email transactional | EmailJS for reminders | Same + Resend or Postmark for receipts/welcome |
| Native packaging | PWA only | + Capacitor (iOS) + Bubblewrap (Android) |

### 6.2 Stripe integration

**Flow:**
1. User clicks "Upgrade to Pro" in Settings
2. App calls `supabase.functions.create-checkout` (new edge function)
3. Edge function creates a Stripe Checkout Session with `client_reference_id = user_id`, returns the URL
4. User completes payment on Stripe's hosted page
5. Stripe sends `checkout.session.completed` webhook to `supabase.functions.stripe-webhook`
6. Webhook updates `users.plan = 'pro'`, `plan_expires_at = period_end`
7. App polls or uses Supabase Realtime to update UI

**Why Stripe Checkout (hosted) instead of Stripe Elements:**
- No PCI compliance burden on us
- Built-in support for Apple Pay, Google Pay, Link
- Subscription management handled by Stripe (customer portal link in Settings)
- Lower engineering cost (days vs weeks)
- 2.9% + 30¢ per transaction (standard Stripe pricing)

**Customer portal:** Settings → "Manage subscription" → opens Stripe customer portal. User can update card, cancel, switch plan. Webhook fires on every change.

### 6.3 Paywall enforcement

**5-free tier:**
- RLS policy: `count(*) from subscriptions where user_id = auth.uid() <= 5` (when `users.plan = 'free'`)
- But RLS can't enforce count limits in a single statement, so use a trigger:
  ```sql
  create or replace function public.check_free_tier_limit()
  returns trigger as $$
  begin
    if (select plan from public.users where id = auth.uid()) = 'free' then
      if (select count(*) from public.subscriptions where user_id = auth.uid()) >= 5 then
        raise exception 'Free tier limit reached. Upgrade to Pro for unlimited subscriptions.';
      end if;
    end if;
    return new;
  end;
  $$ language plpgsql;
  ```
- Frontend shows a paywall modal when user tries to add a 6th sub on Free

**Pro features (full analytics, 12-month projection, custom categories):**
- Gated in the UI: show "Pro feature" badge + upgrade CTA
- For analytics, can enforce server-side via RLS on a new `analytics_aggregates` view

### 6.4 Multi-device sync (Pro only)

- Supabase Realtime subscribes to changes on `subscriptions` table for the current user
- When Pro user opens app on phone + laptop, both stay in sync within ~1 second
- Free users: localStorage only, no realtime
- Conflict resolution: last-modified-wins, with a "merge" UI if both have unsaved changes when sync happens

### 6.5 Native packaging

**Android (Bubblewrap):**
- TWA (Trusted Web Activity) wraps the PWA
- ~1 day of setup, $25 one-time Play Console fee
- App shows up in Play Store, links to `subkit-ten.vercel.app` content
- Push notifications work via web push (no native code needed)

**iOS (Capacitor):**
- Wraps the PWA in a WKWebView
- Requires Mac + Xcode + Apple Developer Account ($99/yr)
- ~1-2 days of setup, but **App Store review can be unpredictable** (Apple has been rejecting PWA-wrappers since EU DMA)
- Backup plan: skip iOS for v1.0, ship Android-only

**Recommendation:** Ship Android via Bubblewrap first. Ship iOS later if Play Store traction is good. Don't risk $99 + 2 weeks of review delay on day 1.

## 9. Risks and unknowns

| Risk | Severity | Mitigation |
|---|---|---|
| Nobody pays | HIGH | Soft launch, validate pricing in 30 days, adjust |
| Apple rejects iOS app | MEDIUM | Skip iOS for v1.0, ship Android only |
| Stripe integration bugs | MEDIUM | Use Stripe Checkout (hosted) not Elements, well-trodden path |
| Free → Pro conversion is < 1% | MEDIUM | A/B test the paywall placement, simplify upgrade flow |
| 5-sub free limit is too restrictive | LOW | Data: most users stay under 5 forever; 5 is industry standard |
| EmailJS rate limits hit us at scale | LOW | Move to Resend/Postmark at 100+ Pro users |
| Supabase costs spike with Pro users | LOW | Free tier covers 500 MB, 2 GB egress; we use < 1% of that |
| User data lock-in (free user can't export) | LOW | Always support CSV/JSON export (already in v0.2.1) |
| Multi-currency bugs in invoices | LOW | Test with 5+ currencies before launch |
| Telegram bot abuse (someone uses our service for spam) | LOW | Not a real concern since the bot is per-user, not shared |
