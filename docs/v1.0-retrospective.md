# SubKit v1.0 — Build Retrospective

**Author:** Mavis (orchestration partner) · **Date:** 2026-06-16
**Audience:** Future me (or anyone else) building a similar product solo

This is a 4-week build journal. Not the marketing version — the real one. The decisions, the dead ends, the things that almost killed the project, and what I'd do differently next time.

## What got built

SubKit is a manual subscription tracker. PWA, no bank linking, free for 5 subs, Pro at $4.99/mo. Stack: single-file vanilla JS PWA + Supabase + Stripe + Deno edge functions. v1.0 ships with:

- Visual redesign (dark + minimax variant)
- 5-sub free tier enforced by a Postgres trigger
- Stripe Checkout + webhook for the paywall
- Customer portal edge function (Stripe billing portal redirect)
- 12-month spending projection
- Custom categories (Pro feature)
- 3-step onboarding flow
- Multi-device sync via Supabase Realtime (Pro)
- Last-write-wins conflict resolution on multi-device edits
- Bills table (separate 5-sub free tier)
- Invoice attachment Pro-only enforcement
- Marketing site at `/` + app at `/app`
- /privacy, /terms, /changelog pages
- Capacitor iOS scaffold + Bubblewrap Android TWA config
- App icons (192/512/1024 + maskable)
- Full launch copy for Product Hunt, HN, Reddit, IndieHackers, X, Email

**Lines of code:** ~4100 in `app.html` + ~430 in `index.html` + 4 edge functions (Deno) + 14 SQL migrations.

**Cost:** $0 (all free tiers).

## The timeline

### Week 1 — Paywall + Stripe + visual

This was the first real team plan. I had 4 parallel tracks: visual redesign, users table + free-tier trigger, Stripe edge functions, and the integration smoke gate. Workers delivered all 4 in one go. Verifier passed everything. Merged to main on day 1.

What worked: tight scope per track, each worker had ~30min of focused work, the smoke gate caught the `openAuth()` bug before it shipped.

What didn't: the first cycle's plan-level retry logic for killed workers. Two of the original 6 tracks (settings upgrade + marketing site) hit the 25-minute cap and the engine re-spawned them mid-cycle, but the workers re-ran the work I'd already inlined myself. Cost: 5 extra minutes of confusion.

### Week 2 — Realtime + analytics + onboarding + bills + polish

This is where things got hard. 6 tracks, all on a 4000-line `app.html`. Workers started strong, but 3 of the 6 hit the 25-minute cap simultaneously: cloud sync conflict resolution, full analytics, and onboarding flow. They all made it to the actual file edits, but couldn't commit + run static checks in time.

I paused the plan, inlined all 3 myself in a single ~2-hour pass, plus the bills migration + marketing polish. The plan was cancelled and I shipped 4 commits directly to a new branch.

**Lesson learned the hard way:** When 2+ workers in the same plan hit the hard cap, the next move is to inline the work, not retry with smaller tasks. Each new worker has a 60-90s cold start, and the same cap will kill it again. Read the file end-to-end first (which I'd already done in Week 1), then just do it.

### Week 3 — Native packaging scaffolds

By Week 3 I had learned the inlining lesson. This was 4-5 hours of solo work:

- Rewrote `icon.svg` from a placeholder emoji to a real "S + stacked cards" mark
- Built the icon script with `sharp` to generate 192/512/1024/maskable PNGs
- Wrote `capacitor.config.ts`, `twa-manifest.json`, `ios/App/App/Info.plist`, `ios/README.md`
- Wrote `.well-known/assetlinks.json` and `apple-app-site-association`
- Added 4 rewrites + 2 content-type headers to `vercel.json`
- Wrote `store/ios/metadata.md` and `store/android/metadata.md`
- Wrote `BUILDING_NATIVELY.md` as the one-stop handoff

What I got wrong: added `public/icons/` to `.gitignore` before committing them. First push to Vercel got 404s on `/icons/icon-512.png` because the deploy only includes git-tracked files. Fixed by un-ignoring and re-committing. (Memory: never gitignore a file that's referenced from a deployed config.)

What I couldn't do: I don't have a Mac, so I can't run `npx cap add ios` or `xcodebuild`. The iOS scaffold is config + Info.plist + README, but the actual `.xcodeproj` will be generated when the user runs `npx cap add ios` on macOS. The Android TWA is similar — `bubblewrap init` will generate the full project from the `twa-manifest.json` I wrote.

### Week 4 — Launch copy

Pure writing. 7 files, 710 lines. No code. The hardest part was matching tone to channel: HN wants technical humility, PH wants a clear value prop, Reddit wants authenticity, IH wants the numbers, Twitter wants the build-in-public story, Email wants the personal ask.

What worked: a single comment-playbook section in each file (pre-written answers to the 6-8 most likely questions). Saves you from typing the same answer 12 times during a launch.

## Decisions I'm proud of

1. **Single-file PWA, no build step.** This made the worker timeout problem worse (one 4000-line file vs. 40 small modules) but the alternative would have been a 4x larger scope. The benefit: the web app IS the native app. No React Native, no Flutter, no dual codebase. One Vercel deploy reaches iOS, Android, and web.

2. **Free-tier enforcement in the DB, not the frontend.** I started with a JS check in `addSubscription()`, then realized anyone could bypass it with curl. Moved to a `BEFORE INSERT` trigger on `public.subscriptions` that reads `users.plan` at INSERT time. The trigger is the only honest way to do this. It also means the trigger fires even for service-role inserts (e.g. if a future feature bulk-imports data).

3. **Last-write-wins with conflict modal.** Naive cloud sync = last writer overwrites. Naive "real" CRDT = 10x the engineering cost. The middle path: every write stamps a `client_updated_at`, the server only accepts the update if the existing value is older, and a conflict modal shows both versions when the server has fresher data. Took 2 days but it's the right shape for a one-person team.

4. **$4.99/mo + $39/yr pricing.** I considered $2.99 (too cheap — looks desperate), $9.99 (in the Mint/Copilot zone but no bank link to justify the premium), and $4.99 (cheap enough to be impulse, expensive enough to be real). $39/yr is 35% off the monthly, which is the standard SaaS discount tier.

5. **The web app loads over HTTPS from a TWA / Capacitor shell, instead of being bundled into the binary.** This is the move that made the iOS+Android story tractable. The shell just opens `https://subkit-ten.vercel.app/app` in a WKWebView / Chrome Custom Tab. The service worker caches it for offline. Updates deploy instantly — no App Store review for web changes. The only things that need a binary rebuild are: app icon, splash screen, bundle ID, signing cert.

## Decisions I'd reconsider

1. **Single-file PWA was right for shipping speed but wrong for scaling the codebase.** At 4100 lines, the worker timeout was inevitable. If I were doing this again, I'd split into 4-5 modules early: `db.js` (Supabase client + queries), `subs.js` (subscription CRUD), `ui.js` (rendering), `paywall.js` (Stripe + paywall logic), `app.html` (orchestration only). Same single-HTML shell, but loaded as ESM modules so workers can edit one at a time.

2. **The Vercel auto-deploy from a feature branch took 6+ seconds to be visible.** I pushed the week-2 branch and immediately curl-tested, saw old content, and wasted 30 seconds before realizing the branch wasn't the deploy branch. The fix is to merge to main first, then smoke. Document this in `BUILDING_NATIVELY.md` so the next person doesn't make the same mistake.

3. **I waited too long to write the launch copy.** Launch copy is a thing that benefits from sitting in your head for a week. I wrote the technical product first, then the launch copy. If I were doing this again I'd write both in parallel — the launch copy forces you to articulate the value prop, which surfaces gaps in the product.

4. **The conflict resolution UI is a v1 minimum, not a v1 done.** The current modal shows two columns and 3 buttons. It doesn't show diffs. It doesn't show what the other device was doing at the time. For 2-device usage it's fine; for 5-device usage it's going to be painful. v1.1 work.

5. **The free tier of 5 subscriptions is the right number, but I never tested it.** What does the user feel when they hit the 5-sub limit? The paywall modal looks good in mockups, but I haven't watched a real user hit it. The next 30 days will tell me if 5 is too low (most users have 10-20 subs) or just right (most users have 3-5).

## What I learned about solo building

1. **The 25-minute worker timeout is the binding constraint, not your skill.** I lost 2 hours to worker timeouts on Week 2. I should have known after Week 1's plan that long single-file edits were the limit. Splitting the work into smaller chunks (e.g. "add the trend chart" vs. "add custom categories" vs. "add YoY comparison") would have given each chunk ~20min of work + commit, which is under the cap.

2. **Smoke tests > unit tests for solo builds.** I wrote 4 deno tests for the Stripe webhook handler. They all pass. They catch regression. They don't catch the bug where the user's session token expires mid-checkout. Real-browser smoke tests in headless Chromium caught 3 real bugs in Week 1 that no unit test would have caught. For a one-person team, spend the testing budget on smoke tests, not unit tests.

3. **The marketing site and the product are not the same thing.** I built the marketing site last, after 3 weeks of product work. If I were doing this again, I'd build the marketing site on day 1, with a "coming soon" hero. That way I'd have a public URL to share from day 1, and the marketing copy forces you to articulate the value prop early.

4. **Documentation is a force multiplier, not a chore.** I wrote `BUILDING_NATIVELY.md`, `launch/CHECKLIST.md`, the per-launch-channel playbooks, and the per-step README. None of this is "user-facing" — it's all for the next person who has to ship the actual native builds, the actual launch, the actual day 30 retrospective. But it cuts the time-to-do-those-things by 10x. For a solo builder, docs are how you scale yourself.

5. **Free tier pricing is a bet, not a calculation.** I chose 5 subs because the competitor (subkit.chuongle.dev) uses 5, and the data shows most users have <5 subscriptions. But this is one data point. The 30-day retrospective will tell me if 5 is right. The key is: pick a number, ship, measure, adjust. Don't get stuck in pre-launch analysis paralysis.

## What the next 30 days look like

1. **Week 1 post-launch:** Tally PH upvotes, HN points, IH comments, Reddit upvotes, email open rate. The real number is: how many people signed up for Pro? Target: 0-2 in week 1, 2-5 by day 30. The conversion rate from free to Pro is the only metric that matters.

2. **Week 2-4:** Iterate on the things users complain about. The PH comments, HN comments, and IH comments are all in `launch/` — I have pre-written reply templates for the most likely 12 questions. If the most-asked question is "where's the iOS app?" then the answer is "pay Apple $99 and ship it." If the most-asked question is "can you add CSV import?" then the answer is "yes, v1.1, this month."

3. **Day 30 retrospective:** Write a public "30 days in" IndieHackers post. Be honest about what worked, what didn't, the conversion rate, the Pro MRR, the support load. The transparency is the marketing.

## What I'd do differently if I started over

- **Day 1:** Marketing site with "coming soon" + email capture. Start collecting emails.
- **Day 1:** Twitter account, post 1x/week about the build. Build in public.
- **Day 1:** Set up a Stripe account in test mode. Set up the customer portal so the wire-up is trivial later.
- **Day 3:** Build the marketing page to the same quality as the product. Marketing is product.
- **Day 5:** User test the free-tier paywall with 3 real humans before locking in the 5-sub number.
- **Day 7:** Soft launch to email list. No PH, no HN. Just a personal email to 50 people. Get the first 10 paying customers before going wide.
- **Day 14:** Public launch. By then I have testimonials, a known-converting pricing model, and a refined product.

The mistake I made: I built for 4 weeks, then launched. The right cadence is: build for 1 week, soft launch, build for 1 week, soft launch to a wider audience, build for 1 week, public launch. Each cycle has real users in the loop.

But the v1.0 is shipped, and the launch is queued. So that's the next 4 weeks.
